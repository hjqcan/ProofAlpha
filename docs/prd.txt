1. 背景与机会
--------------------------------------------------------------------
1.1 背景
- Polymarket 是基于 USDC 的预测市场平台，采用中央限价订单簿（CLOB）撮合 Yes/No（或 Up/Down）合约。
- 合约价格在 0~1 美元之间，可视作事件发生概率的市场表达（例如 0.85 ≈ 85%）。
- 由于 CLOB 的点差、情绪波动、流动性不均衡、临近结算的时间价值等因素，会出现可程序化捕捉的“工程化收益”机会。

1.2 机会来源（来自调研材料的核心结论）
- 两腿套利（Dual-Leg Arbitrage）：在二元合约上分别买入两侧，使两笔成本之和 < 1 美元，结算时赢家一侧兑付恒为 1 美元，从而锁定差价利润；主要风险来自第一条腿敞口时间与极端行情/滑点/延迟。
- 尾盘扫货（Endgame Sweep）：在临近结算且胜率极高（价格接近 1）的市场，赚取“最后几厘钱”；风险为黑天鹅反转与信息误判。
- 流动性做市（Liquidity Making）：在低流动性市场双边挂单赚点差，并可能叠加平台流动性激励；风险为库存偏斜、行情突变与挂单被动成交导致敞口。
- 短线波动/15 分钟市场（Volatility Scalping / “15 分钟心跳”）：利用超短周期市场中的情绪定价误差；对延迟与风控要求更高。

1.3 产品定位
- 这是一个面向开发者/量化交易者的“可扩展、可配置、风控优先”的自动化交易系统，而不是单一策略脚本。
- 采用领域驱动设计（DDD）与分层架构，目标是让策略可插拔、数据与执行可替换、风险可统一治理。

2. 目标与非目标
--------------------------------------------------------------------
2.1 产品目标（Goals）
- G1：提供稳定连接 Polymarket 的数据订阅与交易执行能力（HTTP + WebSocket），支持重连、限流、重试。
- G2：以策略引擎为核心，支持多策略并行调度与统一资金/风险约束。
- G3：提供“可审计”的交易与决策记录（结构化日志 + 关键事件落库），便于复盘与排错。
- G4：支持“模拟/纸面交易（Paper Trading）”与“实盘（Live）”的显式切换，降低上线风险。
- G5：满足低延迟与高可用的工程约束（异步 I/O、故障恢复、状态持久化）。

2.2 非目标（Non-Goals）
- NG1：不承诺盈利或提供投资建议；系统只实现策略执行与风控框架。
- NG2：不在 v1 内实现复杂的机器学习预测或外部新闻智能分析（可预留接口）。
- NG3：不在 v1 内实现完整的 Web UI（可选项），默认以 CLI + 日志 +（可选）简单监控为主。
- NG4：不在 v1 内实现跨交易所对冲（可作为后续扩展）。

3. 用户与使用场景
--------------------------------------------------------------------
3.1 目标用户
- U1：独立开发者/量化交易者：希望快速上线策略、能控制风险、可复盘。
- U2：工程化运营者：关注稳定性、告警、可恢复与可观测性。

3.2 典型场景（User Stories）
- S1：我想启动“两腿套利”，系统自动筛选符合条件的市场并在满足阈值时下单，且单市场投入不超过 X%。
- S2：我想先用模拟模式跑 24 小时，确认策略触发频率、滑点假设与风险指标，再切换为实盘。
- S3：我想在运行中调整阈值（例如 PairCostThreshold），并立即生效且留痕。
- S4：我想随时查看：当前连接状态、订阅市场数、未完成订单数、仓位、PnL、风控触发次数。
- S5：当 WebSocket 断线或 API 报错，系统能自动退避重连并报警；必要时触发“全局熔断”停止下单。

4. 术语与定义
--------------------------------------------------------------------
- 市场（Market）：Polymarket 的一个事件合约市场，包含 marketId、到期时间、状态等。
- 合约/方向（Outcome/Side）：Yes/No 或 Up/Down 两个互斥结果。
- 订单簿（OrderBook）：市场的买卖档深度与最优价等。
- 两腿套利（Dual-Leg）：先买一侧，再买另一侧完成对冲，使成本之和 <= 阈值（目标 < 1）。
- 敞口（Exposure）：未被对冲的单边头寸风险。
- FAK/FOK/GTC/GTD：订单时效类型（以官方最新文档为准；本 PRD 要求系统具备可扩展支持）。
- Paper Trading：仅记录决策与“模拟成交”，不向真实 API 发单。

5. 产品范围与版本规划
--------------------------------------------------------------------
5.1 MVP（可交付最小版本）
MVP 的原则：先把“数据 + 执行 + 风控 + 可观测”打稳，再扩展策略数量与 UI。

MVP 必含能力：
- 数据：市场列表拉取 + 指定市场订单簿订阅 + 用户成交/订单事件订阅（若可用）。
- 执行：必须支持 Live 实盘下单/撤单/查询订单状态（最小闭环）；同时保留 PaperTrading 作为上线前验证模式。
- 风控：单市场资金上限、最大未完成订单数、全局熔断、策略级启停。
- 策略：两腿套利（Dual-Leg Arbitrage）作为 MVP 首发策略（必含）。
- 交互：CLI（启动/停止、查看状态、调整参数、导出日志）。
- 持久化：PostgreSQL（必选）记录订单、成交、仓位、策略状态、风控事件、配置版本与运行状态（便于恢复与审计）。

5.2 V1（增强版）
- 多策略并行（Dual-Leg + Endgame + Liquidity Making 至少 2~3 个）。
- 更完善的策略参数管理与热更新（配置版本化）。
- 监控与告警（指标 + 可选 Prometheus 导出）。

5.3 V2（可选）
- Web/桌面 GUI（Blazor/WPF）用于监控与配置。
- 更高级的回测/仿真框架；外部行情（RTDS/交易所）融合。

6. 功能需求（Functional Requirements）
--------------------------------------------------------------------
说明：本节用“模块 -> 需求点 -> 验收要点”的形式，便于直接拆任务与写测试。

6.1 策略引擎（Strategy Engine）
6.1.1 策略接口与生命周期
- FR-SE-001：提供统一策略接口 ITradingStrategy（至少包含：SelectMarkets / EvaluateEntry / EvaluateExit / OnOrderFilled 或等价回调）。
- FR-SE-002：支持策略生命周期：初始化 -> 运行 -> 暂停 -> 停止 ->（可选）恢复。
- FR-SE-003：策略必须可配置（阈值、市场过滤条件、资金上限、订单类型偏好等），配置变更需留痕。
验收要点：
- 可在 CLI 中启停任意策略，且状态变更可在日志与状态查询中看到。

6.1.2 策略调度与并发
- FR-SE-010：StrategyManager 支持多策略并发运行（至少异步任务并行），并统一协调资金分配与风险限制。
- FR-SE-011：策略间不得互相阻塞；单策略异常不得导致全局崩溃（需隔离与监督）。
验收要点：
- 人为制造单策略异常（抛错/断线），系统仍可维持其他策略运行或安全降级。

6.2 市场数据服务（Market Data）
6.2.1 市场发现与过滤
- FR-MD-001：通过 HTTP 拉取市场元数据（“Gamma API/Markets API”等来源以官方为准），支持定时刷新与增量更新。
- FR-MD-002：提供可复用的市场过滤能力：按到期时间、类别（如 15 分钟市场）、流动性/交易量阈值等筛选。
验收要点：
- 可输出候选市场列表（数量、筛选原因、关键字段）。

6.2.2 实时订单簿与行情订阅
- FR-MD-010：支持通过 WebSocket 订阅订单簿/报价更新，并在本地维护 OrderBook 状态（最优买卖、深度档位等）。
- FR-MD-011：具备断线重连、心跳与退避策略；重连后能恢复订阅与本地状态。
- FR-MD-012（可选）：支持 RTDS 或外部加密货币价格流订阅，用于短线策略。
验收要点：
- 在网络中断后可自动恢复订阅；恢复时间与重试节奏可配置并有日志。

6.3 交易执行与订单管理（Execution）
6.3.1 API 客户端能力
- FR-EX-001：封装 PolymarketApiClient，至少支持：下单、撤单、查询订单、查询余额/仓位（以可用接口为准）。
- FR-EX-002：支持订单时效类型的扩展（FAK/FOK/GTC/GTD 等），并允许策略选择偏好。
- FR-EX-003：支持批量下单（若官方支持；调研材料提及批量限制 15 条，需要配置化与降级策略）。
验收要点：
- 在模拟环境/沙盒（若无则用 mock）可跑通“下单->成交回报/查询->仓位更新”闭环。

6.3.2 限流、重试与幂等
- FR-EX-010：实现请求限流与队列，遵守平台速率限制（调研材料提及 POST /order 在 10 秒窗口存在上限；具体以官方文档为准）。
- FR-EX-011：对可重试错误实现指数退避；对不可重试错误快速失败并上报。
- FR-EX-012：对“重复下单/重复回报”具备幂等处理（订单 ID/客户端生成 requestId）。
验收要点：
- 压测下单请求不应触发平台封禁风险；限流生效可观察（指标/日志）。

6.4 账户/仓位与领域模型（Trading Domain）
6.4.1 核心领域对象
- FR-TD-001：Account、Position、Order、TradingAccount 等模型需显式表达并封装不变量（例如：余额不足禁止下单）。
- FR-TD-002：OrderService 负责下单/撤单编排，并通过领域事件（OrderPlaced/OrderFilled/PositionClosed 等）驱动状态变更。
验收要点：
- 单元测试覆盖核心不变量与状态机（订单状态、仓位均价、已实现/未实现 PnL 计算）。

6.5 风险管理（Risk Management）
6.5.0 风控到底是什么意思（白话版）
- 核心一句话：风控=给机器人加“护栏”，避免因为市场极端情况、执行延迟、部分成交、断线、Bug 等原因，把本来“小赚稳定”的策略变成“偶尔爆仓式大亏”。
- 为什么两腿套利特别需要风控：
  - 两腿套利的理论前提是“买齐两边且总成本 < 1”，但现实会遇到：第一腿成交了、第二腿迟迟买不到（或要用更贵价格才能买到） => 这段时间你是裸露单边风险（敞口）。
  - 另外还有：下单失败/重复下单、部分成交、盘口突然变薄、网络断线、API 变更/限流等“工程风险”。
- 因此风控的目标不是“让你不亏”，而是：
  - 把“单次最坏情况损失”限制在你可接受的范围内
  - 让系统在不确定时自动降级（暂停下单/撤单/退出），并把原因记录下来

6.5.1 风险参数
- FR-RM-001：支持全局与策略级风险参数：
  - 单市场最大资金占用比例（MaxCapitalPerMarket）
  - 单策略最大资金占用比例（MaxCapitalPerStrategy）
  - 全局最大资金使用比例（MaxTotalCapitalUtilization）
  - 最大未完成订单数（MaxOpenOrders）
  - 单策略最大并发市场数（MaxConcurrentMarketsPerStrategy）
  -（可选）当日最大亏损熔断（DailyLossLimitPct）
  -（可选）连续下单失败/错误率熔断（MaxConsecutiveOrderErrors / MaxErrorRate）
- FR-RM-002：提供“全局熔断（Kill Switch）”：触发后立即停止新下单，按策略配置决定是否撤单/平仓。
- FR-RM-003：所有风控触发必须写入“风控事件日志”（含触发原因、当时关键指标、采取的动作），并可导出审计。
验收要点：
- 人为触发阈值后，系统按预期阻止下单并记录触发原因。

6.5.1.1 风控参数建议默认值（保守起步，可配置）
说明：你说“风控不太懂”，我先给一套“不会激进”的默认值，保证先把系统跑稳。后续你观察实际触发频率与收益，再逐步放宽。
- MaxCapitalPerMarket = 0.05（单市场最多动用 5% 资金，避免单市场黑天鹅/敞口把账户打穿）
- MaxCapitalPerStrategy = 0.30（两腿套利策略最多动用 30% 资金；剩余资金作为缓冲与未来扩展）
- MaxTotalCapitalUtilization = 0.50（全局最多用掉 50% 可用资金，避免“满仓运行”导致不可控）
- MaxOpenOrders = 20（避免挂太多单导致管理复杂、也避免误触高频/限流）
- MaxConcurrentMarketsPerStrategy = 3（两腿套利同时最多做 3 个市场，先求稳）
- DailyLossLimitPct = 0.02（可选：当日亏损达到 2% 自动熔断停机；你若更激进可调大）
- MaxConsecutiveOrderErrors = 10（可选：连续 10 次下单失败则熔断，避免 API/签名/网络异常时越错越下）
（已确认：接受上述默认值）

6.5.2 两腿套利特有风险控制
- FR-RM-010：对第一条腿敞口设置时间上限/价格偏离上限；超出则执行预案（撤单/对冲/止损退出）。
- FR-RM-011：对“第一腿敞口金额”设置硬上限（例如不超过总资金的某个比例），防止单边风险堆积。
- FR-RM-012：对“市场剩余时间”设置下限（例如剩余时间过短不允许开第一腿），避免来不及完成对冲。
- FR-RM-013：对“盘口流动性”设置下限（例如买一/卖一深度不足则不允许开仓），避免第二腿无流动性导致高价对冲。
验收要点：
- 在模拟场景中，第一腿成交但第二腿迟迟不满足阈值时，风险预案必定被执行。

6.5.2.1 两腿套利特有风控参数建议默认值（保守起步，可配置）
说明：这部分是“两腿套利”最关键的护栏，用来控制“第一腿成交后未对冲”的最坏情况。
- MaxUnhedgedCapitalPerMarket = 0.02（单市场“未对冲敞口”最多占用 2% 总资金；对冲完成后才允许用到 MaxCapitalPerMarket 的 5%）
- MaxFirstLegExposureSeconds = 120（第一腿成交后，等待第二腿满足 PairCostThreshold 的最长时间；超时必须执行预案：撤单/止损退出/强制对冲）
（已确认：接受上述默认值）

6.6 模拟/纸面交易（Paper Trading）
- FR-PT-001：系统必须支持 PaperTradingMode，所有策略/引擎逻辑保持一致，仅将“发单”替换为“模拟撮合/记录意图”。
- FR-PT-002：模拟成交规则需可配置（例如：按盘口最优价成交、考虑滑点、部分成交概率等）；默认给出简单版本。
验收要点：
- 同一策略在 Paper 与 Live 的行为差异仅限于执行通道；决策日志字段保持一致。

6.7 配置、密钥与运行管理
- FR-CFG-001：支持 appsettings.json + 环境变量覆盖；策略参数支持分层与热更新（至少运行中修改后重启生效，V1 目标热更新）。
- FR-CFG-002：密钥/私钥不得写入代码仓库；日志必须脱敏。
验收要点：
- 在日志中搜索不得出现私钥、API Key 明文。

6.8 日志、审计与可观测性
- FR-OBS-001：结构化日志：至少记录市场、策略、决策原因、阈值、订单参数、成交回报、异常栈。
- FR-OBS-002：关键指标：连接状态、订阅数、下单数、撤单数、成交数、失败数、风控触发次数、延迟分布。
- FR-OBS-003：支持审计导出：按时间导出订单/成交/决策记录（CSV/JSON）。
验收要点：
- 能基于日志与导出文件复现一次交易决策链路。

6.9 命令行界面（CLI）
- FR-CLI-001：提供命令：
  - start/stop：启动/停止某策略或全局
  - status：查看系统/连接/订阅/策略状态
  - positions/orders/pnl：查看仓位、订单、盈亏
  - config get/set：读取/修改关键参数（带确认与留痕）
  - export：导出审计数据
- FR-CLI-002：交互需清晰一致，输出字段固定，便于脚本化调用。
验收要点：
- 通过 CLI 可完成“启动 -> 运行 -> 调参 -> 查看 -> 停止 -> 导出”的全流程。

6.10 可选 UI / API
- FR-UI-001（可选）：Blazor/WPF 仪表盘：市场/订单簿/仓位/风险指标可视化，支持启停与参数调整。
- FR-API-001（可选）：REST/gRPC 对外接口，供远程监控与自动化运维。

7. 非功能需求（Non-Functional Requirements）
--------------------------------------------------------------------
7.1 性能与延迟
- NFR-PERF-001：数据订阅与策略决策链路需异步化；避免阻塞 I/O。
- NFR-PERF-002：关键路径（行情更新 -> 评估 -> 发单）需可观测延迟，并支持优化（内存缓存、最小分配）。
- NFR-PERF-003：建议部署在低延迟网络环境（靠近平台基础设施的 VPS）。

7.2 稳定性与容错
- NFR-REL-001：WebSocket 断线自动重连（退避 + 最大重试策略），恢复后自动重新订阅。
- NFR-REL-002：关键状态（仓位、未完成订单、策略运行状态）需持久化，崩溃后可恢复。

7.3 安全
- NFR-SEC-001：私钥/密钥只允许通过安全配置注入；严禁输出到日志。
- NFR-SEC-002：对外接口（若提供）需鉴权与最小权限原则。

7.4 可维护性与扩展性
- NFR-MAINT-001：遵循 DDD 分层与限界上下文：Trading / MarketData / Strategy / Infrastructure / Application。
- NFR-MAINT-002：新增策略只需实现接口并注册到 StrategyManager，不应改动核心执行链路。

====================================================================
8. 合规与风控边界（Compliance & Guardrails）
--------------------------------------------------------------------
- COM-001：系统需允许配置“地理/KYC 限制开关与校验逻辑”（具体规则以平台与当地法律为准）。
- COM-002：默认启用审计日志与关键动作留痕，便于合规与复盘。
- COM-003：明确“禁止行为”提示（例如：异常高频、疑似操纵市场的参数组合可触发内部告警/阻断）。

10. 验收标准（Acceptance Criteria）
--------------------------------------------------------------------
10.1 MVP 验收（最小可用）
- AC-001：可在 PaperTrading 模式持续运行 >= 6 小时无崩溃，断线可恢复订阅。
- AC-002：两腿套利策略可产生可解释决策日志，且能走通“发单/撤单/状态更新（模拟）”闭环。
- AC-003：风控阈值触发时能阻断下单并记录原因；Kill Switch 可手动触发。
- AC-004：CLI 支持启停策略、查看状态、导出审计数据。
- AC-005：Live 模式可用：在小额资金下完成真实下单/撤单，并能正确更新订单状态与仓位（以官方可用接口为准）；出现下单失败/断线时能自动降级并留痕。

10.2 V1 验收（增强版）
- AC-101：支持至少 2 个策略并行运行（例如 Dual-Leg + Endgame 或 Dual-Leg + Liquidity），且 StrategyManager 可进行资金与风险协调。
- AC-102：两腿套利在“第一腿敞口超时”的场景下能按预案处理（撤单/对冲/止损）。
- AC-103：指标与告警可覆盖连接/下单失败/风控触发等关键事件。

11. 风险清单与对策
--------------------------------------------------------------------
11.1 市场与策略风险
- R-STR-001：第一腿敞口导致净亏损（极端波动/滑点/成交延迟）
  - 对策：敞口超时/偏离阈值强制处理；限制单市场资金；优先选择流动性更好的市场
- R-STR-002：尾盘策略黑天鹅反转
  - 对策：单市场上限 + 止损阈值 + 更严格的临近结算筛选
- R-STR-003：做市库存偏斜
  - 对策：库存上限/再平衡逻辑；动态调整报价；必要时撤单退出

11.2 工程与平台风险
- R-ENG-001：API 变更、限流策略更新
  - 对策：客户端封装与版本适配；速率限制配置化；关键接口契约测试
- R-ENG-002：断线/网络抖动导致状态不一致
  - 对策：事件驱动 + 幂等 + 状态持久化；重连后对账（拉取未完成订单/余额）
- R-ENG-003：密钥泄露
  - 对策：密钥只进环境变量/安全存储；日志脱敏；最小权限；定期轮换
