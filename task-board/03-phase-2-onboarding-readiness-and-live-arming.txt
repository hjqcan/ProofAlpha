Phase 2: Onboarding, Readiness, and Live Arming
===============================================

Status
------
[DONE]


Intent
------
Turn first run and Live enablement into an explicit operator workflow. The user
should know what is configured, what is unsafe, and what remains blocked before
the system can place real orders.


Core Boundary
-------------
Readiness and Live arming are gates, not recommendations. Live order placement
must remain blocked unless configuration, compliance, credentials, account sync,
and risk posture are all acceptable.


Accepted Scope
--------------
- first-run readiness wizard
- credential and configuration diagnostics without secret leakage
- Live arming checklist
- Paper default path
- readiness UI and CLI/API parity


Out of Scope
------------
- exchange account creation
- KYC workflow automation
- storing secrets in the browser
- weakening existing compliance blocks


Acceptance Criteria
-------------------
1. First-run diagnostics show database, API reachability, WebSocket, background
   jobs, account sync, compliance, risk limits, and execution mode.
2. Live arming requires explicit operator action and records evidence.
3. Missing or unsafe Live prerequisites block command execution with actionable
   reasons.
4. Paper mode remains usable when Live prerequisites are missing.
5. CLI, API, and Web App present consistent readiness outcomes.
