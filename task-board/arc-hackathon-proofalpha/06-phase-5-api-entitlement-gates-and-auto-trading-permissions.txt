Phase 5: API Entitlement Gates and Auto-Trading Permissions
===========================================================

Status
------
[TODO]


Intent
------
Make subscription state enforce real product behavior: unsubscribed users cannot
access full opportunities or request auto-trading permission.


Core Boundary
-------------
Entitlement checks are product access checks. They do not replace risk,
compliance, kill switch, strategy lifecycle, or Live arming gates.


Accepted Scope
--------------
- Access-gated signal detail endpoint.
- Access-gated opportunity detail endpoint.
- Auto-trading permission request model.
- Paper-only auto-trading demo gate.
- Audit of blocked and accepted access decisions.


Out of Scope
------------
- Real custody or user fund management.
- Live trading enablement without existing Live arming.
- Direct wallet signing in backend.


Permission Model
----------------
`ViewSignals`:

- can list public signal summaries
- can view full opportunity details if active

`ViewReasoning`:

- can view evidence/reasoning hashes and summaries
- never sees raw private prompts or secrets

`RequestPaperAutoTrade`:

- can request Paper strategy start for allowed strategy key
- still requires strategy enabled and risk not blocked

`RequestLiveAutoTrade`:

- can request Live strategy start only if all existing Live gates pass
- probably disabled for hackathon unless explicitly demonstrated safely

[TODO] AH-P5-T005 Add auto-trading permission command.

Command shape:

```powershell
autotrade arc autotrade request --wallet <address> --strategy two_leg_arbitrage --mode Paper
```

Behavior:

- checks `RequestPaperAutoTrade`
- checks strategy exists
- checks command mode allows mutation
- starts/resumes strategy through existing Strategy command path
